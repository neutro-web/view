# Checkpoint v1.0.0 — Hardening + Surpassing

> **Meaning:** everything from 0.5.0→1.0.0 hardens and surpasses. Performance, fault tolerance,
> efficiency, ergonomics, security, reliability — plus features/improvements that **measurably beat
> the status quo**, not match it.
> **Bar:** a very high bar. Measured margin over incumbents, not feature presence.

**Status:** NOT STARTED (gated on v0.5.0). **Reconciled to** `007eaa2`.

This checkpoint is intentionally a **structured frame, not a fixed task list** — its contents firm up
as v0.1.0 benchmark baselines and v0.5.0 parity reveal where nv actually has margin to push. Filling
it speculatively now would violate "don't rush to an end state." The axes below are the dimensions;
specific items get added when evidence (benchmarks, profiles, parity gaps) names them.

---

## The six axes

### A1 — Performance (measured margin over incumbents)
The v0.1.0 benchmark baseline (CP-2c) is the denominator. v1.0.0 performance items are those where
nv can *beat* Solid/Svelte/vanilla on a measured op, not match. Candidate territory (confirm against
baseline, do not assume):
- **Event delegation** (PT-5) — bundle + per-element listener cost.
- **`handlerKind: 'stable'` skip-the-effect** — already a named hook in the IR (`EventBinding`;
  v0 always emits `'reactive'` wrapper-effect, the `'stable'` path is deferred to benchmark
  validation per §10). A stable handler skips the per-event wrapper effect. **Concrete A1 item with
  an existing seam** — validate against the v0.1.0 benchmark baseline before enabling (soundness:
  misclassification must cost perf, never correctness).
- **LIS/move-minimization** — closed-by-tradeoff 2026-06-22; reconcile already moves nodes via
  `insertBefore` (O(1) single-move). Reopens **only** on a *measured* reorder/swap-throughput deficit
  at scale (real-app or benchmark), not by assumption. Currently no such deficit is in evidence.
- **kind-split** — `ReactiveNode` width; accepted structural gap, evidence-tested CLEAR. Reopens
  only on real-app frame-budget breach. *Trigger stands; do not manufacture.*
- **Compiler specialization depth** — the compiler-skips-only-provable-work axiom leaves headroom;
  measure where erasure/specialization buys real frame time.

### A2 — Fault tolerance
- `errorBoundary` exists **in core** but the **renderer does not yet wrap mounts/effects/list-item
  bodies in boundaries** (core-only today). Real gap, not just hardening: a throwing effect or
  render body currently has no renderer-level containment. Wire renderer boundaries, then harden
  against the full failure surface (effect throws, async rejection once PT-1 lands, render-time
  errors, disposal during error).
- Adversarial disposal / leaked-edge guarantees (the POC already proves zero leaked edges on clean
  dispose; extend to error paths).

### A3 — Efficiency (memory, bundle, GC)
- Memory benchmarks from js-framework-benchmark (ready/run/update/clear memory, leak detection).
- Tree-shakeable exports; per-entry-point bundle cost vs incumbents.
- Construction-cost (createSignals ~5–7× structural gap is known; tied to kind-split trigger).

### A4 — Ergonomics
- Authoring surface polish (informed by the v0.1.0 clone + benchmark app — where did authoring hurt?).
- Error messages / diagnostics quality (the compiler has a diagnostics channel; surface quality).
- Dev loop: HMR, dev server, error overlay (deferred from v0.1.0; a real DX gap for adoption).
- Two-way binding ergonomics, `<Index>`/`<Switch>` authoring (from PT-3).

### A5 — Security
- **Default value insertion is XSS-safe by construction** — dynamic text/child values go through
  `createTextNode` (interpreter L188/L411), never `innerHTML`; the only `innerHTML` is on the
  parser-produced static `shape.html`, not interpolated user values. State and protect this as a
  guarantee. The audit is at the **boundaries where it could break**:
  - the deferred `ChildBinding` DOM-Node/`TemplateIR` insertion path (IR notes it as designed-but-
    deferred) — must not become a raw-HTML escape hatch without explicit opt-in + docs;
  - attribute/prop injection (e.g. `href="javascript:"`, event-handler attributes);
  - any future raw-HTML API — must be explicit, named, and documented as unsafe (Solid's
    `innerHTML` precedent).
- `$style` injection safety (CSS-custom-property lowering already avoids re-running factories;
  confirm no value-driven selector injection).
- Untrusted-content boundaries if SSR (PT-6) lands (hydration mismatch / injection).

### A6 — Reliability
- Cross-browser conformance (the §12 conformance suite + Playwright ×3 as the standing gate).
- API stability commitment (1.0 = semver stability promise; contract freeze discipline).
- Real-app soak: a HN/Twitter clone run as a living integration test, not a one-shot.

---

## Features that must *measurably beat* status quo (not match)

Empty by design. Items land here only when a v0.1.0/v0.5.0 measurement or parity gap shows nv has a
*specific, measured* advantage to press — e.g. "nv's run-once + compiler-erasure does X in N ms where
Solid does N+k." A feature that merely matches the incumbent belongs in v0.5.0, not here.

---

## Reaching v1.0.0

v1.0.0 is reached when: A1–A6 each have their evidence-named items DONE or consciously deferred;
the API/contract is frozen with a stability promise; the real-app soak test is green cross-browser;
and at least the headline performance/efficiency margins are *measured and published* vs incumbents.
