# Checkpoint v0.5.0 ‚Äî Feature Parity

> **Meaning:** nv expresses what the incumbents express, **filtered through nv's axioms**. Parity
> with what nv's architecture *can and should* do ‚Äî not 1:1 API mirroring.
> **Bar:** parity, not superiority. Surpassing is v1.0.0.

**Status:** NOT STARTED (gated on v0.1.0). **Reconciled to** `007eaa2`.

Grounded against the current incumbent surface (Solid as the architectural sibling ‚Äî run-once, no
VDOM, fine-grained; Svelte 5 runes converging on the same model). Each parity target is filtered
through nv's locked axioms; some incumbent features are inapplicable, subsumed, or must take a
different shape. See the axiom-filter table in [`../ROADMAP.md`](../ROADMAP.md).

> **Closure guardrail (applies to every target below):** the four reactive primitives are a **closed
> set**. No parity target may add a new reactive *graph primitive*. Async, store, and context are
> built **by composition over the four** or as **out-of-graph utilities** (the `pubsub` precedent).
> Any proposal to add a graph primitive is a **reversal of the primitive-closure axiom** and must be
> logged as such ‚Äî it is never a routine "parity feature." This protects the static guarantees of
> `derived` (purity), `sync`, and `pubsub` that the compiler's soundness depends on.

## Parity targets (each needs an axiom-shape ruling before it's scoped)

### PT-1 ‚Äî Async data: read-centric async‚Üísignal construct
**Status: OPEN QUESTION (design-open) ‚Äî AXIOM-CONSTRAINED.**
nv has **no async primitive**. Incumbents: Solid `createResource` / async signals (read-centric ‚Äî
pause only when data isn't there to render).

**Axiom constraint (load-bearing ‚Äî do not violate):** the four reactive primitives are a **closed
set**, and the contract establishes the pattern that non-pure, non-graph concerns sit **outside** the
graph as utilities (the `pubsub` "fifth helper outside the graph" note, contract ¬ß2; external-source
protocol ¬ß8.6).
Therefore async parity **must** be built as **composition over the existing four primitives** (a
`signal` holding `{data, loading, error}` driven by an `effect` that consumes an external async
source) and/or as an **external-source utility** feeding `sync` ‚Äî exactly the established
pubsub‚Üísync external-producer path. It **must NOT** be added as a new reactive graph node/primitive,
and it must not widen `derived` purity or `sync`/`pubsub` static guarantees.

**Architect ruling required:** confirm the composition/external-utility shape and that no graph
primitive is added. If a ruling ever proposes a new graph primitive, that is a **reversal of the
primitive-closure axiom** and must be logged as such, not slipped in as a parity feature.
This likely gates Suspense (v1.0.0).

### PT-2 ‚Äî Nested reactive state (store-equivalent)
**Status: OPEN QUESTION (design-open) ‚Äî AXIOM-CONSTRAINED.**
Incumbents: Solid `createStore` (proxy, per-property tracking). Parity = nested reactive state that
preserves fine-grained semantics.

**The contract already anticipates this** (¬ß2 close: "the renderer and the `store` proxy primitive
are built on top, outside this contract"). So a store is sanctioned *as a built-on-top API primitive*,
explicitly **outside** the reactive-core graph ‚Äî which matches the constraints below; it is not a new
graph node.

**Axiom constraints (load-bearing):**
1. **Primitive closure** ‚Äî a store is **composition over signals** (built on top, outside the
   reactive-core contract per ¬ß2), **not** a new reactive graph primitive. Same closure rule as PT-1.
2. **Granularity** ‚Äî it must preserve per-leaf fine-grained tracking; it must not collapse a nested
   object into one coarse signal that re-runs consumers on any sub-change (that would set back the
   performance axiom ‚Äî the whole point of fine-grained reactivity).
3. **DOM-free core** ‚Äî it lives outside core (renderer-adjacent or standalone), built on `signal`.

**Open shape questions for the architect ruling:** proxy-based (per-property signal under the hood,
Solid-style) vs path-based; how disposal/ownership composes. **Must not dissolve signal granularity
or the closure.**

### PT-3 ‚Äî Control flow completion
**Status: PARTIAL.**
- `<Show>`-equivalent: **HAVE** (`conditional`).
- `<For>`-equivalent (keyed, by-identity): **HAVE** (`list`).
- `<Index>`-equivalent (positional tracking): **DONE** ‚Äî `<recycle>` landed `3b00064` (2026-06-29). Distinct construct (not an `<each>` mode); position-identity, node-recycling via Op-3 re-bind. Node-churn = 0/scroll-step and real-browser focus-follows-position both verified. See decision-log recycling entries.
- `<Switch>`/`<Match>` (multi-branch): **GAP** ‚Äî today single conditional; multi-branch is a new
  IR shape or a documented nested-conditional pattern. Architect: shape ruling.

### PT-4 ‚Äî Context / provide-inject
**Status: OPEN QUESTION (design-open).**
Ownership graph (`getOwner`, `currentOwner`) exists. Parity = a context API building on ownership
(provide at a node, inject in descendants). **Closure-safe by construction:** this is an
owner-graph *traversal utility*, not a new reactive primitive ‚Äî it reads the existing ownership tree,
adds no node kind. Must avoid a global registry (would break the no-VDOM/owner-scoped model). Likely
tractable given the owner graph already tracks the tree.

### PT-5 ‚Äî Event delegation
**Status: PARTIAL (correctness done, perf gap).**
Direct `addEventListener` works (correct). Incumbents delegate common events to the document root
for bundle/perf wins. Parity-for-perf, not correctness. May defer to v1.0.0 perf work.

### PT-6 ‚Äî SSR / hydration
**Status: GAP (scope decision needed).**
No server path today. **Axiom note:** the DOM-free core is an asset ‚Äî the reactive graph makes no DOM
assumptions, so it runs identically server-side. The constraint: SSR string-rendering and hydration
**must live entirely in the renderer/a server entry point**, never in core ‚Äî the core stays DOM-free
and agnostic (locked, 2026-06-24). The parity target is `renderToString` + hydrate (view-engine
scope), **not** a meta-framework. **Scope question:** v0.5.0 parity requirement or v1.0.0 item?
**Architect ruling on scope + the renderer-only shape.**

### PT-7 ‚Äî Two-way binding
**Status: DONE (`sync`).**
`sync` is nv's single reactive‚Üísignal-write construct; `:value`/`:checked` landed. Parity met.

### PT-8 ‚Äî Scoped styles
**Status: DONE (`$style`).**
Slot domain closed; only `$style √ó <each>` browser cascade leg deferred (v0.1.0 fold-in).

## Explicitly OUT of parity scope (through v1.0.0)

- **Router / meta-framework.** nv is a view engine. Parity = a documented signals+conditional
  routing *pattern*, not a built-in router. (Incumbents put this in SolidStart/SvelteKit.)
- **Primitive ecosystem** (solid-primitives-equivalent). Community layer, not engine parity.

## Reaching v0.5.0

v0.5.0 is reached when PT-1..PT-4 have architect axiom-shape rulings AND are DONE (or consciously
deferred with trigger), PT-3 control flow is complete, and the SSR scope question (PT-6) is decided
(in or out). PT-5/PT-7/PT-8 are done-or-perf-deferred.

**The first v0.5.0 action is not coding ‚Äî it's the axiom-shape rulings** for PT-1, PT-2, PT-4, PT-6.
Those are decision-log entries. Until they exist, the parity targets are hypotheses, not scoped work.
