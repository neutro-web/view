## Throw-safety check (commission's "confirm, don't assume")

Read the restored function directly (`src/renderer/interpreter.ts:777-849`): the effect body reads `const P = pool.length` fresh at the start of every run (line 789) and uses `P` as the single source of truth for the rebind/grow/shrink split ‚Äî there is no separate `activeCount`-style bookkeeping variable anywhere in the dispose path. HWM's pre-`0ac39f9` bug was specifically a drift between such a counter and the real pool state in HWM's reuse/grow loop; that variable simply does not exist here, so that bug class cannot occur by construction ‚Äî there's nothing to desync `pool.length` from. In the grow loop (lines 802-830), if `itemTemplate`/`mountFragment`/the multi-root check throws, `pool.push(...)` for that iteration never executes, so `pool.length` stays exactly at the count of iterations that actually succeeded ‚Äî consistent with what was mounted. The one residual risk is pre-existing and not B‚Ä≤-specific: if the throw happens after `mountFragment` has already inserted DOM nodes but before that iteration's `pool.push()` runs (e.g. the "Multi-root list items are not supported" check at line 816-820 fires after insertion), that iteration's DOM could be orphaned. This is the same pattern shared by `wireList`'s Op-1 growth path and was already known/out of scope going into this revert ‚Äî it is not introduced or worsened by restoring the dispose-on-shrink body, and no fix or new test for it is in scope here.
